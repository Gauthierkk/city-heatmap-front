## Decision #10 details

**Category model.** One category is active at a time; switching it resets the type-filter pills to the new category's full tag set, clears the focused store and any load error, but keeps the entered address and geocoded marker so closest-place results re-rank automatically.

**Food split (5 / 13).** The existing 18 food types are divided without removing any: Grocery = supermarket, convenience, greengrocer, organic, frozen_food (5 types the user would visit for everyday shopping); Specialty food = the other 13, including bakery (destination-retail trades). The split is client-side — both categories share the single `stores-<city>.geojson` file, so no extra fetch is needed when switching between them.

**Fitness subtypes.** Six types, all service-only (no `shop` tag in OSM data allowed by the normaliser): gym, yoga, pilates, martial_arts (covers karate/judo/taekwondo/boxing/mma), dance, climbing. Each meets the ≥30-stores-in-one-city bar: Overpass counts Paris / NYC / Austin — fitness_centre 293/635/109, yoga 46/87/13, pilates 9/54/3, martial-arts regex 50/83/5, dance 97/82/14, climbing 30/12/7. Crossfit (2/3/2) folds into gym. The normalisation priority (yoga > pilates > martial_arts > climbing > dance > gym) resolves multi-sport venues deterministically; a generic `sports_centre` without a matching sport tag returns null (excluded).

**French title frame.** The FR title pattern `'{category} à {city}'` was chosen because it preserves the existing default phrase "Commerces alimentaires à Paris" verbatim for the Grocery category. Future categories must fit the `"<noun phrase> à <ville>"` frame — if a future label doesn't fit naturally in French, the template should be revisited then, not now.

**Per-dataset overwrite guards.** The food guard (100 features) is unchanged from the original single-dataset script. The fitness guard is 50 — Austin fitness totals ≈150 features, well above 50, but the food threshold of 100 would be too tight a safety margin for a smaller dataset. Both guards block a clobber on partial/empty Overpass responses.

**Category switcher UI.** The category name is a second inline `<select>` in the panel title, matching the existing city select in style. `titleSegments(lang)` drives both selects from a single template, so adding a fourth category requires only a new `CategoryDef` entry and a new data file — no UI code changes.

## Implementation notes

- Distance-field overlay: adaptive cells over the city bbox (Paris 50 m ≈ 70 k cells, NYC 125 m ≈ 149 k cells). Stores are bucketed into a 500 m spatial grid; nearest-neighbour search expands ring-by-ring, early-exiting once the ring's minimum possible distance exceeds the current best. Equirectangular distance approximation (m/° at the city's mid-latitude) replaces haversine per cell. Result rendered to a `<canvas>`, exported as a PNG data-URL, and added as a MapLibre `image` source displayed via a `raster` layer.
- The colour ramp is user-configurable from the "Heatmap settings" panel: "Red within" (10–50 m, default 50 m) and "Blue beyond" (100–500 m, default 500 m); intermediate stops scale proportionally between the two. Recompute is debounced 250 ms since the field costs ~100–300 ms and sliders fire continuously while dragging.
- The overlay is clipped to the city's administrative boundary (`npm run fetch-boundary -- <city>` writes `public/data/boundary-<city>.geojson`, Douglas-Peucker-simplified, area sanity-checked per city: Paris ~105 km²; NYC ~1,223 km² because the OSM admin polygon extends into harbour water). Clipping is a `destination-in` composite fill of the boundary path on the field canvas (even-odd rule handles holes), leaving the per-cell compute unchanged. A thin `line` layer outlines the boundary so the clip edge reads as intentional. If the boundary file is missing or invalid, the overlay renders unclipped.
- Nominatim's usage policy disallows true keystroke autocomplete; suggestions are debounced (800 ms, min 3 chars) to stay well under 1 req/s. Fine for local use; revisit before public hosting.
- Type filters call `setData` with a filtered feature collection; the distance-field overlay recomputes on filter or ramp-bound change.
